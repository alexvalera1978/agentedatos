1. Qué es Orus

Orus es un agente de datos de negocio que responde en lenguaje natural a preguntas sobre los datos de una empresa (ventas, stock, clientes, márgenes, etc.). El usuario escribe una pregunta como lo haría a una persona ("¿cuánto hemos vendido en junio?") y Orus consulta las fuentes de datos reales, calcula la respuesta y la presenta con un texto resumen, una tabla y, si procede, un gráfico.

Está diseñada como plataforma multi-cliente y multi-fuente: la misma aplicación sirve a distintos clientes, cada uno con sus propios orígenes de datos (un ERP, una tienda Shopify, un Excel…) y sus propias reglas de negocio, sin tocar el código.

Idea clave. Orus no tiene las tablas ni las reglas "cableadas" en el programa. Cada cliente se configura como datos (a qué se conecta) más una guía (cómo interpretar sus datos). Así, dar de alta un cliente nuevo es configurarlo, no programarlo.

2. Arquitectura general

La aplicación tiene tres capas y un motor de inteligencia:

CapaTecnologíaFunción
Frontend (interfaz)React + ViteEl chat donde se pregunta y el panel de administración donde se dan de alta clientes y fuentes.
Backend (servidor)Node.js + ExpressLa API, el motor del agente y los conectores a las fuentes de datos.
Inteligencia (LLM)OpenAI (GPT‑4o)Interpreta la pregunta y decide qué datos consultar mediante "herramientas" (tool‑calling).
ConectoresMódulos reutilizablesUn conector por tipo de origen: ERP, Shopify, Excel, REST, SQL. El mismo conector sirve a cualquier cliente de ese tipo.

Flujo de una pregunta (paso a paso)

  1. El usuario escribe la pregunta en el chat y elige el cliente activo.
  2. El backend carga la configuración de ese cliente (sus fuentes + su guía).
  3. El LLM recibe la pregunta + las reglas (prompt general + guía del cliente) y decide qué herramientas usar.
  4. Las herramientas consultan las fuentes reales (ERP, Shopify…) y devuelven datos.
  5. El LLM redacta el texto resumen; el frontend muestra la tabla, el gráfico y la etiqueta de fuente.

3. Multi-cliente (multi-tenant)

Cada cliente es un "tenant": una configuración independiente con sus fuentes, su guía y sus opciones. Los tenants están aislados entre sí — cada cliente solo ve sus propios datos.

Cada tenant se guarda como un archivo de datos (JSON), no como código. Esto permite crear o editar un cliente en caliente, desde el panel, sin reprogramar ni redesplegar.

4. Los dos niveles de "prompt" (instrucciones)

Orus combina dos capas de instrucciones en cada pregunta:

a) Prompt general (fijo, igual para todos los clientes)

Son las reglas de comportamiento globales, escritas en el código. Definen cómo responde el agente independientemente del cliente: responder en español, no inventar cifras, usar nombres y no códigos, formato de euros europeo, no negarse a consultar por fechas recientes, aplicar filtros de forma consistente, evitar filas duplicadas, etc.

Dónde server/llm/openai-agent.js → función systemPrompt(...).

b) Guía por cliente (editable por cada cliente)

Son las reglas de negocio específicas de cada cliente: qué tablas usar, cómo se relacionan, qué excluir, qué fuente usar para cada tipo de pregunta. Se edita desde el panel, sin tocar código.

Dónde campo prompt del tenant (server/data/tenants/<id>.json), editable en el panel de administración. Para el cliente actual, el texto de referencia está en scripts/guide-smt2.txt.

Regla de reparto. La configuración de la fuente dice dónde están los datos (URL, credenciales, tablas). La guía dice cuándo usar cada fuente y cómo combinarlas. Con esto, un mismo motor sirve a cualquier cliente.

5. Herramientas del agente (tool-calling)

El LLM no accede a los datos directamente: usa herramientas que el backend le ofrece. Elige cuáles llamar según la pregunta. Las herramientas disponibles dependen de las fuentes que tenga el cliente.

7. Funciones destacadas

  • Gráficos automáticos. Cuando el cliente lo tiene activado, elige el gráfico que mejor encaja (líneas para evolución temporal, barras/horizontales para rankings, tarta para pocas categorías, comparativa para dos años). Dónde client/src/Chart.jsx.
  • Memoria de conversación. Recuerda los últimos intercambios para preguntas de seguimiento ("¿y por meses?").
  • Aprendizaje ("recordar"). Cuando le corriges algo reutilizable, lo guarda en la guía de ese cliente para el futuro.
  • Etiqueta de fuente. Cada respuesta muestra de dónde salieron los datos: 🛍️ Shopify o 🗄️ ERP (o ambas en consultas cruzadas), y si respondió la IA.
  • Formato europeo. Importes en euros (1.326,80 €), fechas dd/mm/aaaa, meses con nombre.

8. Panel de administración

Permite gestionar todo sin tocar código:

  • Crear clientes.
  • Conectar varias fuentes por cliente (añadir, editar, borrar): ERP, Shopify, Excel…
  • Auto-descubrir tablas/columnas y (opcional) emparejar campos.
  • Escribir la guía del agente (las reglas de negocio).
  • Activar los gráficos.

Dónde client/src/Admin.jsx.

9. Despliegue y entornos

EntornoUso
DesarrolloDonde se programan y prueban los cambios.
ProducciónDonde el cliente usa la aplicación (expuesta por ngrok mientras no haya dominio propio).

El código se sincroniza entre entornos mediante Git: se sube desde desarrollo (git push) y se baja en producción (git pull). El arranque unificado (npm start) compila el frontend y sirve la interfaz y la API desde un único puerto.

10. Seguridad

  • Secretos fuera del código. Claves de OpenAI, ERP y Shopify viven en un archivo .env local (no se sube a Git). En la configuración de cada cliente se referencian como variables, nunca en texto plano.
  • SQL de solo lectura. El agente solo puede ejecutar consultas SELECT; se bloquea cualquier escritura o borrado.
  • Aislamiento por cliente. Cada tenant accede únicamente a sus fuentes.
